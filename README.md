# Cowrie SSH Honeypot Monitor

![Python](https://img.shields.io/badge/python-3.10%2B-blue?logo=python&logoColor=white)
![License](https://img.shields.io/badge/license-MIT-green)
![Platform](https://img.shields.io/badge/platform-Ubuntu%2022.04-orange?logo=ubuntu&logoColor=white)

A self-contained SSH honeypot system built on [Cowrie](https://cowrie.readthedocs.io) that captures real-world attack sessions and displays them on a custom web dashboard. Designed to be deployed on any Linux VM with a public IP — cloud VPS deployments (e.g. Microsoft Azure) are strongly recommended for meaningful attack traffic.

---

## Architecture

```
Attacker
   │
   │  SSH (port 2222)
   ▼
┌─────────────────────┐
│   Cowrie Honeypot   │  ← fake filesystem regenerated on every start
└────────┬────────────┘
         │ appends to
         ▼
   cowrie.json  (JSONL)
         │
         │  cowrie-split.path watches for changes
         ▼
┌─────────────────────┐
│  session_parser.py  │  ← triggered automatically by systemd path unit
└────────┬────────────┘
         │ writes to
         ▼
    honeypot.db  (SQLite)
         │
         │  Flask reads on every request
         ▼
┌─────────────────────┐
│    dashboard.py     │
└────────┬────────────┘
         │  HTTP (port 5000)
         ▼
      Browser
```

## 1. Prerequisites

| Requirement | Details |
|---|---|
| OS | Ubuntu 22.04 LTS (other Debian-based distros may work) |
| Python | 3.10 or newer |
| RAM | 1 GB minimum |
| Disk | 10 GB minimum |
| Network | Public-facing IP address — a cloud VM is strongly recommended |

> **Why a public IP?**  
> Cowrie is only interesting when it receives real internet traffic. On a local machine or private network, you will capture nothing. Within hours of exposure on a cloud VM, automated scanners and credential-stuffing bots will begin connecting.

## 2. Cowrie Installation

Follow the [official Cowrie documentation](https://cowrie.readthedocs.io) to perform a standard installation. The quick-start summary is below.

### 2a. Install system dependencies

```bash
sudo apt-get update
sudo apt-get install -y git python3-virtualenv libssl-dev libffi-dev \
    build-essential libpython3-dev python3-minimal authbind virtualenv
```

### 2b. Create a dedicated user

Cowrie should **not** run as root. Create a dedicated system user:

```bash
sudo adduser --disabled-password cowrie
sudo su - cowrie
```

All remaining steps in this section run as the `cowrie` user.

### 2c. Clone and install Cowrie

```bash
git clone https://github.com/cowrie/cowrie ~/cowrie
cd ~/cowrie
python3 -m venv cowrie-env
source cowrie-env/bin/activate
pip install --upgrade pip
pip install -r requirements.txt
```

### 2d. Configure `cowrie.cfg`

Copy the sample config and edit it:

```bash
cp ~/cowrie/etc/cowrie.cfg.dist ~/cowrie/etc/cowrie.cfg
nano ~/cowrie/etc/cowrie.cfg
```

Apply the following changes:

**Listen on port 2222 (no root required):**
```ini
[honeypot]
listen_endpoints = tcp:2222:interface=0.0.0.0
```

**Enable TTY session recording:**
```ini
[honeypot]
ttylog = true
ttylog_path = var/lib/cowrie/tty
```

**Point the filesystem at the custom pickle generated by this project:**
```ini
[shell]
filesystem = /home/cowrie/cowrie/share/cowrie/custom.pickle
```

**Enable JSON logging and point it at this project's log directory:**
```ini
[output_jsonlog]
enabled = true
logfile = /home/cowrie/honeypot/logs/raw-logs/cowrie.json
```

> Create the log directory first if it does not exist:
> ```bash
> mkdir -p ~/honeypot/logs/raw-logs
> ```

### 2e. Configure `userdb.txt`

Cowrie uses `etc/userdb.txt` to decide which credential combinations are accepted.
The following configuration accepts a handful of plausible accounts while also
accepting any credentials — which ensures automated scanners always get in and
generate interesting session data:

```
# etc/userdb.txt
root:x:!root
root:x:!123456
root:x:*
deploy:x:deploy123
deploy:x:*
ubuntu:x:ubuntu
ubuntu:x:*
admin:x:admin123
admin:x:*
tomcat:x:*
oracle:x:*
*:x:somepassword
*:x:*
```

Lines starting with `!` **reject** that specific password for the given user.
A bare `*` on either side is a wildcard. The effect here is: reject the most
common root passwords (so those probes are logged as failures), but accept
everything else.

```bash
nano ~/cowrie/etc/userdb.txt
```

### 2f. Port redirect (optional but recommended)

Attackers probe port **22**, not 2222. To capture that traffic without running
Cowrie as root, add an `iptables` redirect:

```bash
sudo iptables -t nat -A PREROUTING -p tcp --dport 22 -j REDIRECT --to-port 2222
sudo iptables -t nat -A OUTPUT -p tcp --dport 22 -o lo -j REDIRECT --to-port 2222
```

To persist across reboots:

```bash
sudo apt-get install -y iptables-persistent
sudo netfilter-persistent save
```

## 3. Project Setup

All steps below run as the `cowrie` user unless stated otherwise.

### 3a. Clone the repository

```bash
git clone <your-repo-url> ~/honeypot
```

The resulting layout should be:

```
~/honeypot/
├── dashboard/
│   ├── dashboard.py        # Flask routes
│   ├── db.py               # SQLite query layer
│   ├── threat.py           # Threat scoring engine
│   ├── session_parser.py   # JSONL → SQLite parser
│   ├── generate_fs.py      # Fake filesystem generator
│   ├── signatures.json     # Detection rules
│   └── templates/          # Jinja2 HTML templates
├── logs/
│   ├── raw-logs/           # cowrie.json written here by Cowrie
│   └── analysis/           # honeypot.db and CSV exports
├── scripts/
│   ├── start_cowrie.sh     # Generates FS, builds pickle, starts Cowrie
│   ├── stop_cowrie.sh      # Graceful shutdown
│   └── split_cowrie.sh     # Runs session_parser.py
└── systemd/                # Reference unit files (copied by setup.sh)
```

### 3b. Create the Python virtual environment

```bash
cd ~/honeypot/dashboard
python3 -m venv .venv
.venv/bin/pip install --upgrade pip
.venv/bin/pip install -r requirements.txt
```

The `requirements.txt` pins exact versions for reproducibility:

```
Flask==3.1.3
Flask-HTTPAuth==4.8.1
python-dotenv==1.2.2
Werkzeug==3.1.8
```

### 3c. Configure dashboard credentials

The dashboard is protected by HTTP Basic Auth. Create a `.env` file in the
`dashboard/` directory:

```bash
nano ~/honeypot/dashboard/.env
```

```ini
DASHBOARD_USER=admin
DASHBOARD_PASS=your_secure_password_here
```

### 3d. Install systemd units

The `systemd/` directory contains reference unit files. The units are written
with absolute paths for the `cowrie` user (`/home/cowrie/...`). Install them:

```bash
sudo cp ~/honeypot/systemd/*.service /etc/systemd/system/
sudo cp ~/honeypot/systemd/*.path    /etc/systemd/system/
sudo systemctl daemon-reload
```

> If your username is **not** `cowrie`, edit each unit file and replace
> `/home/cowrie/` with the correct home directory before copying.

### 3e. Make scripts executable

```bash
chmod +x ~/honeypot/scripts/*.sh
```

## 4. systemd Services

This project uses four systemd units that work together as a pipeline.

### Unit overview

| Unit | Type | Purpose |
|---|---|---|
| `cowrie.service` | forking | Starts/stops Cowrie; regenerates the fake filesystem on every start |
| `honeypot-dashboard.service` | simple | Serves the Flask dashboard on port 5000 |
| `cowrie-split.service` | oneshot | Parses `cowrie.json` into `honeypot.db` |
| `cowrie-split.path` | path | Watches `cowrie.json` for changes; triggers `cowrie-split.service` automatically |

### How they connect

```
cowrie-split.path
      │  watches cowrie.json
      │  on change, activates →
      ▼
cowrie-split.service
      │  runs split_cowrie.sh
      │  which calls session_parser.py
      ▼
honeypot.db  (updated)
      │
      ▼
honeypot-dashboard.service  (reads DB on each request)
```

### `cowrie.service`

Managed by `start_cowrie.sh` and `stop_cowrie.sh`. On every start, the script:

1. Runs `generate_fs.py` to produce a fresh randomized fake filesystem
2. Runs Cowrie's `createfs` tool to compile it into a pickle file
3. Starts Cowrie

The `ExecStopPost` directive cleans up any stale PID file automatically on stop.

### `honeypot-dashboard.service`

Runs the Flask app directly via the virtualenv Python interpreter, so no system-level
Flask installation is needed. Restarts automatically if it crashes.

### `cowrie-split.service` + `cowrie-split.path`

`cowrie-split.path` watches `/home/cowrie/honeypot/logs/raw-logs/cowrie.json` for
modification events. Whenever Cowrie writes a new event, systemd fires
`cowrie-split.service`, which re-parses the entire log into a fresh SQLite database.
The parser deduplicates events using `(uuid, eventid, timestamp)` keys, so re-parsing
is always safe and idempotent.

### Enable and start all services

```bash
sudo systemctl enable cowrie.service honeypot-dashboard.service cowrie-split.path
sudo systemctl start  cowrie.service honeypot-dashboard.service cowrie-split.path
```

### Verify they are running

```bash
sudo systemctl status cowrie.service
sudo systemctl status honeypot-dashboard.service
sudo systemctl status cowrie-split.path
```

### Useful service commands

```bash
# View live Cowrie output
sudo journalctl -u cowrie.service -f

# View dashboard logs
sudo journalctl -u honeypot-dashboard.service -f

# Manually trigger a database rebuild
sudo systemctl start cowrie-split.service

# Restart everything
sudo systemctl restart cowrie.service honeypot-dashboard.service
```

## 5. Fake Filesystem

Every time Cowrie starts, `generate_fs.py` generates a randomized but internally
consistent fake server environment. This makes the honeypot far more convincing to
attackers who probe the filesystem for credentials and configuration.

### What is generated

The fake server simulates a multi-user Ubuntu production machine with four interactive
accounts and two system accounts:

| Account | Role | Notable files |
|---|---|---|
| `root` | Superuser | `passwords.txt`, `notes.txt`, `.bash_history`, `.ssh/id_rsa` |
| `deploy` | Deployment service account | `.env` with DB credentials and AWS keys, `projects/<repo>/`, `scripts/backup.sh` |
| `ubuntu` | Cloud VM sysadmin | `notes.txt` with infrastructure IPs, `.bash_history` |
| `admin` | Senior sysadmin | `notes.txt` with credential list, `audit.sh`, `.bash_history` |
| `mysql` | System account (no shell) | `etc/mysql/my.cnf` with plaintext credentials |
| `www-data` | Web server (no shell) | `var/www/html/config.php` with DB credentials |

### Cross-referenced state

All accounts reference the **same** randomly generated values: hostname, internal IPs,
database name, database credentials, and AWS key pair. This means an attacker who
pivots from `/root/passwords.txt` to `/home/deploy/projects/<repo>/.env` will find
credentials that match, making the environment feel genuinely real.

### Running it manually

```bash
# Default: Cowrie installed at ~/cowrie
python3 ~/honeypot/dashboard/generate_fs.py

# Custom Cowrie location
python3 ~/honeypot/dashboard/generate_fs.py --cowrie-dir /opt/cowrie
```

After generation, the script outputs the randomly chosen hostname, repo name, DB
server IP, and contact emails so you can verify the world is internally consistent:

```
Cowrie Filesystem Generator
========================================
  Hostname   : web-prod-01
  Repo       : api-service
  DB name    : production_db
  DB server  : 10.0.0.147
  Backup     : 10.0.0.83
  Admin      : james.smith@company.internal
  Deploy     : sarah.jones@company.internal

  Written to : /tmp/cowrie-src
  Written to : /home/cowrie/cowrie/honeyfs
```

## 6. Dashboard Features

Access the dashboard at `http://<your-ip>:5000` after starting the services.

### Overview page (`/`)

- Total session count, unique attacker IPs, auth attempts, successful logins, and sessions where commands were run
- Daily attack timeline chart
- Top 5 attempted usernames and passwords
- HASSH SSH client fingerprint breakdown (useful for identifying scanner tooling)

### Sessions page (`/sessions`)

- Paginated, filterable table of all captured sessions (25 per page)
- Filter by date range, threat label (LOW / MEDIUM / HIGH / CRITICAL), or search by IP/session ID
- Each row shows: source IP, timestamp, duration, login result, command count, and threat score badge

### Session detail page (`/session/<id>`)

- Full chronological event timeline (connect → auth → commands → disconnect)
- All authentication attempts for the session (username, password, success/failure)
- Threat detections with matched rule names and triggering commands
- Link to TTY replay if a terminal log was recorded

### TTY Replay (`/replay/<id>`)

Frame-accurate terminal replay of attacker sessions using [xterm.js](https://xtermjs.org/).
Replays the raw ttyrec binary at real-time speed, including all control sequences
(colour, cursor movement, clearing). Sessions are only replayable if Cowrie recorded
a TTY log (`ttylog` column is populated in the database).

### IP Map (`/map`)

Geolocates all attacker IPs and plots them on a world map with session counts.
Requires that connections are arriving from real public internet IPs — local or
NAT'd traffic will not geolocate meaningfully.

### Session export

Every session detail page includes an **Export JSON** link that downloads a structured
JSON file containing the session metadata, full event list, auth attempts, and threat
result. Useful for offline analysis or incident documentation.

## 7. Threat Scoring

Each session is scored by `threat.py` using a rule-based engine that reads
detection signatures from `signatures.json`.

### How scoring works

1. Every command in the session is matched against all regex patterns in `signatures.json`
2. Each matched rule contributes a fixed number of points based on severity
3. Two additional behavioural checks run on top of signature matching
4. The total is capped at 100 and mapped to a label

### Severity weights

| Severity | Points |
|---|---|
| CRITICAL | 40 |
| HIGH | 20 |
| MEDIUM | 8 |
| BEHAVIORAL | 15 |

### Score labels

| Score range | Label |
|---|---|
| 75 – 100 | 🔴 CRITICAL |
| 50 – 74 | 🟠 HIGH |
| 25 – 49 | 🟡 MEDIUM |
| 0 – 24 | 🟢 LOW |

### Signature categories

**CRITICAL** — Attacker is actively hostile and covering tracks:
- Reverse shells (Bash TCP, netcat, Python)
- Base64-encoded command execution
- Clearing shell history or wiping system logs
- Reading `/etc/shadow` directly
- Injecting SSH keys into `authorized_keys`

**HIGH** — Privilege escalation and persistence attempts:
- Enumerating sudo privileges (`sudo -l`)
- Searching for SUID binaries
- Downloading scripts to `/tmp/`
- Known exploit tool names (linpeas, linenum, dirtycow, pwnkit, dirty_pipe)
- Establishing cron persistence
- Executing payloads from `/tmp/`, `/dev/shm/`, `/var/tmp/`
- Reading credential files (`.env`, `passwords`, `credentials`, `secrets`)
- Lateral movement via SSH
- Data exfiltration (scp, rsync, curl upload)

**MEDIUM** — Reconnaissance (often automated scanners):
- Kernel and OS fingerprinting (`uname -a`, `/etc/os-release`)
- User identity checks (`whoami`, `id`, `groups`)
- Network enumeration (`ifconfig`, `ip addr`, `netstat`, `ss`)
- Process listing (`ps aux`)
- Filesystem hunting (`find / -name`)
- SSH config enumeration (`~/.ssh/known_hosts`, `~/.ssh/config`)

### Behavioural detection

In addition to signature matching, the scorer applies two behavioural rules:

**Command burst** (`BEHAVIORAL`, 15 pts): 10 or more commands issued within a 1-second
window indicates an automated script pasting a payload rather than a human typing.

**Recon sequence** (`BEHAVIORAL`, 15 pts): 3 or more distinct reconnaissance rule hits
in a single session flags systematic enumeration, even if each individual command would
only score MEDIUM on its own.

Each rule fires **at most once per session** to prevent artificial inflation from
repeated identical commands.

## 8. Network Requirements

| Port | Service | Direction |
|---|---|---|
| 2222 | Cowrie SSH honeypot | Inbound from anywhere |
| 5000 | Flask dashboard | Inbound — restrict to trusted IPs in production |
| 22 | Your real SSH management access | Inbound from your IP only |

> **Security note:** Port 5000 is protected by HTTP Basic Auth, but for production
> deployments consider restricting it to your own IP in the firewall, or placing it
> behind a reverse proxy with HTTPS.

If you applied the optional `iptables` redirect from port 22 → 2222, ensure your
firewall still allows inbound traffic on port 22 (the redirect happens at the kernel
level, after the firewall pass).

## 9. Deployment on Microsoft Azure

### 9a. Create the VM

1. In the Azure portal, create a new **Virtual Machine**
2. Image: **Ubuntu Server 22.04 LTS**
3. Size: **B1s** (1 vCPU, 1 GB RAM) is sufficient to start; scale up if you see high CPU from Cowrie
4. Authentication: SSH public key (recommended)
5. Note the public IP address assigned to the VM

### 9b. Add Network Security Group inbound rules

In the Azure portal, navigate to your VM → **Networking** → **Add inbound port rule**.
Add the following rules:

| Rule name | Port | Protocol | Source | Action |
|---|---|---|---|---|
| Allow-SSH-Honeypot | 2222 | TCP | Any | Allow |
| Allow-Dashboard | 5000 | TCP | Your IP | Allow |
| Allow-SSH-Admin | 22 | TCP | Your IP | Allow |

> Set the Dashboard rule source to **your own IP address** rather than `Any` to
> prevent the dashboard from being publicly accessible.

### 9c. Connect and set up

```bash
ssh -i ~/.ssh/your_key.pem azureuser@<public-ip>

# Become the cowrie user once it exists
sudo su - cowrie
```

Follow the steps in sections 2 and 3 of this README to complete installation.

### 9d. Access the dashboard

Once the services are running, open a browser and navigate to:

```
http://<public-ip>:5000
```

Login with the credentials you set in `dashboard/.env`.

Within a few hours of the VM being reachable on the internet, Cowrie will begin
receiving automated SSH probes. The dashboard will populate as `session_parser.py`
processes the incoming events.

## 10. Troubleshooting

### Cowrie fails to start — PID file exists

If Cowrie was not stopped cleanly, a stale PID file may prevent it from starting:

```bash
rm ~/cowrie/var/run/cowrie.pid
sudo systemctl start cowrie.service
```

The `ExecStopPost` in `cowrie.service` removes this file automatically on a clean stop,
but it can remain after a crash.

### Dashboard is not updating with new sessions

Check that the path watcher and parser service are active:

```bash
sudo systemctl status cowrie-split.path
sudo systemctl status cowrie-split.service

# Manually trigger a rebuild to confirm the parser works
sudo systemctl start cowrie-split.service
sudo journalctl -u cowrie-split.service -n 50
```

If the path unit shows `inactive`, make sure it was enabled:

```bash
sudo systemctl enable --now cowrie-split.path
```

### TTY replay is blank

The replay page requires Cowrie to have recorded a terminal log for that session.
Check whether the session has a value in the `ttylog` column:

```bash
sqlite3 ~/honeypot/logs/analysis/honeypot.db \
  "SELECT session_id, ttylog FROM sessions WHERE ttylog IS NOT NULL LIMIT 5;"
```

If `ttylog` is always `NULL`, verify that `cowrie.cfg` has TTY logging enabled:

```ini
[honeypot]
ttylog = true
ttylog_path = var/lib/cowrie/tty
```

Then restart Cowrie and wait for a new session to arrive.

### No IPs appearing on the map

The geolocation map only shows meaningful data for **public** IP addresses. If all
sessions show `127.0.0.1` or private RFC 1918 addresses (`10.x`, `192.168.x`,
`172.16–31.x`), either:

- The VM is behind a NAT and not directly reachable from the internet, or
- The firewall or NSG is blocking inbound connections on port 2222

Verify that port 2222 is open:

```bash
# From another machine
nc -zv <public-ip> 2222
```

### Cowrie starts but no sessions appear in the dashboard

Check that the `cowrie.json` log file is being written to the correct path:

```bash
tail -f ~/honeypot/logs/raw-logs/cowrie.json
```

If the file is empty or missing, verify the `[output_jsonlog]` section in `cowrie.cfg`
points to the correct absolute path. Then check that the directory is writable by the
`cowrie` user:

```bash
ls -la ~/honeypot/logs/raw-logs/
```

### Dashboard returns 401 Unauthorized

Confirm that `dashboard/.env` exists and contains `DASHBOARD_USER` and `DASHBOARD_PASS`:

```bash
cat ~/honeypot/dashboard/.env
```

Then restart the dashboard service to pick up any changes:

```bash
sudo systemctl restart honeypot-dashboard.service
```
